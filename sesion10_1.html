Monolito vs. Microservicios

Arquitectura Monolítica

Analogía: Un Rascacielos "Todo en Uno".

Toda la aplicación (interfaz, lógica de negocio, acceso a datos) es una única unidad indivisible. Si una parte falla, todo el edificio puede colapsar.

Pros:
  • Simple de empezar y desarrollar.
  • Fácil de probar (todo en un solo lugar).
  • Despliegue sencillo.
Contras:
  • Difícil de escalar (si necesitas más potencia en "usuarios", debes escalar todo el rascacielos).
  • Tecnología "atrapada".
  • Frágil: un error tumba todo.

Arquitectura de Microservicios

Analogía: Un Campus de Edificios Especializados.

La aplicación se divide en servicios pequeños e independientes (un edificio para "usuarios", otro para "oportunidades", otro para "notificaciones").

Pros:
  • Escalabilidad independiente (puedes añadir 10 pisos solo al edificio de "usuarios").
  • Libertad tecnológica (cada edificio con su mejor tecnología).
  • Resiliencia: si un edificio falla, el resto del campus sigue funcionando.
Contras:
  • Complejidad operacional (muchas partes que gestionar).
  • La comunicación entre servicios es más compleja.

Interactivo: La Analogía del Camarero

Una API es un contrato, un intermediario. En el restaurante del software, el Camarero es la API. No entras a la cocina a prepararte la comida; le das una orden clara al camarero y él se encarga del resto.

1. Cliente (Tu App)

2. Camarero (La API)

Recibe órdenes del Cliente y las traduce para la Cocina. Trae de vuelta el plato.

3. Cocina (El Servidor)

Recibe peticiones de la API, prepara la data (el plato) y lo entrega.

Solución Guiada: Diseño de API de YachayConnect

Esta es la solución a la Ficha de Trabajo 10: el diseño de la API REST para nuestro recurso principal, Oportunidad, basado en nuestro Diagrama de Clases.

GET /api/v1/oportunidades

Obtiene una lista de todas las oportunidades.

Respuesta (200 OK):

[
    { "id": 1, "titulo": "Practicante Frontend", "empresa": "TecCorp", "estado": "Abierta" },
    { "id": 2, "titulo": "Analista de QA Jr", "empresa": "DataAnalytica", "estado": "Abierta" }
]

GET /api/v1/oportunidades/{id}

Obtiene los detalles de una oportunidad específica.

Respuesta (200 OK):

{
    "id": 1,
    "titulo": "Practicante Frontend",
    "descripcion": "Buscamos un estudiante apasionado por la tecnología...",
    "empresa": { "id": 10, "nombre": "TecCorp" },
    "estado": "Abierta",
    "etiquetas": ["JavaScript", "React", "CSS"]
}

POST /api/v1/oportunidades

Crea una nueva oportunidad (usado por Reclutadores validados).

Cuerpo de Petición (Request Body):

{
    "titulo": "Practicante Backend",
    "descripcion": "Se requiere conocimiento en Java y Spring Boot.",
    "tipoContrato": "Pasantia",
    "etiquetas": ["Java", "Spring", "SQL"]
}

Respuesta (201 Created):

{
    "id": 3,
    "titulo": "Practicante Backend",
    ...
}

PUT /api/v1/oportunidades/{id}

Actualiza una oportunidad existente.

Cuerpo de Petición (Request Body):

{
    "titulo": "Practicante Backend (Actualizado)",
    "descripcion": "Se requiere conocimiento en Java y Spring Boot.",
    "estado": "Cerrada",
    "etiquetas": ["Java", "Spring", "SQL", "AWS"]
}

DELETE /api/v1/oportunidades/{id}

Elimina (o archiva) una oportunidad.

Respuesta (204 No Content):

(Sin cuerpo de respuesta)
